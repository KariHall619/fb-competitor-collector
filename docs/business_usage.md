# FB 竞品采集 Skill 使用方式

这套工具给业务同学在 Codex 里用自然语言操作，不需要记脚本命令。

## 使用前准备

1. 用平时工作的 Chrome 登录 Facebook。
2. 在同一个 Chrome profile 里启用 OpenCLI Browser Bridge。
3. Codex 会优先通过 OpenCLI 打开需要采集的 Facebook 主页；如果自动打开失败，再按提示手动确认目标主页能看到真实帖子列表。
4. 飞书 CLI 需要保持用户身份登录。

## 常用说法

检查工具状态：

```text
使用 FB 竞品采集 skill，检查一下现在能不能用。
```

检查会确认飞书登录、输出表配置、用户身份限制、OpenCLI Browser Bridge 是否可用。

测试目标页面是否能读到正文：

```text
使用 FB 竞品采集 skill，试一下目标 Facebook 页面能不能抓到帖子正文，不要写飞书。
```

采集目标页面：

```text
使用 FB 竞品采集 skill，采集目标 Facebook 页面里可见的帖子，先保存到内容库。
```

扩量时的快速采集会先把候选保存到 SQLite，再由后台补全任务继续补精确时间、评论/回复引流链接、互动数据、帖子类型、文章材料和中文概要。正式飞书表是严格最终输出：普通 `--sync` 只写入通过当前质量门的 `ready_for_output` 记录；待补抓候选留在本地库和补抓队列里，不写入正式表。

采集竞品调研账户第一个账号：

```text
使用 FB 竞品采集 skill，采集竞品调研账户第一个账号今天的帖子，并同步到飞书。
```

导入已有抓取结果：

```text
使用 FB 竞品采集 skill，把这份抓取结果导入内容库。
文件路径是：<这里填文件路径>
```

筛选某一天竞品帖子：

```text
使用 FB 竞品采集 skill，筛选 5 月 21 日的竞品帖子，并写入飞书筛选结果。
```

筛选高浏览量帖子：

```text
使用 FB 竞品采集 skill，筛选浏览量大于 10 万的帖子。
```

筛选高点赞文本帖：

```text
使用 FB 竞品采集 skill，筛选点赞量超过 100 的文本帖。
```

## 业务侧需要提供什么

采集时，Codex 会优先通过 OpenCLI 打开或匹配目标账号页，并把得到的 `--tab-page` 传给子任务，避免占用用户当前正在看的 Chrome 标签页。详情补抓会尽量复用同一个详情页标签处理多条候选；如果需要回退，也只应使用自动化打开并跟踪的标签页，不应打断用户正在使用的其他窗口。

手动导入时，至少需要提供以下三列：

```text
帖子链接
文章链接
简述
```

如果有这些字段，也可以一起提供：

```text
账号名
账号主页链接
账号类型：竞品账号 / 内部主页账号
发帖时间
帖子类型
浏览量
点赞量
```

## 重要说明

- 只有 OpenCLI Browser Bridge 路线用于实时采集。
- 如果只是 OpenCLI daemon 未运行，系统可通过环境检查的 `--fix-opencli` 自动尝试启动/恢复；如果 daemon 已运行但 Chrome 扩展没有连接到业务 Chrome profile，才需要人工修复插件或 profile 设置。
- Facebook 采集标签页必须使用同一个已登录 Chrome profile，并能看到帖子列表。
- 如果采集标签页出现登录入口、游客预览、只显示一条预览帖子、空白动态壳、评论片段或没有真实帖子，系统会立即停止并提示 `human_intervention_required`，需要人工登录/确认页面后再重试。
- 重复的帖子链接不会重复入库。
- 有效 Facebook 候选先入 SQLite；缺精确时间、评论/回复引流链接、文章摘要或互动数据时标记为 `needs_enrichment`，后续继续补采。质量门禁只限制最终飞书同步，不阻止候选入库。
- 没有浏览量/点赞量时不会阻塞入库，会标记为“互动数据未确认”；但如果业务要求按互动数据筛选，缺失互动数据的记录不会命中。
- 发帖时间只要求精确到小时，格式为 `2026年5月19日 17:00`。
- `19min`、`1h`、`3h`、`12h`、`1d` 这类相对时间只作为主页候选范围线索，不再换算成正式发帖时间。正式 `posted_at` 必须来自 Facebook 的精确时间悬停提示或页面 DOM 里的精确时间属性。
- 时间悬停由 Skill 自动完成：优先使用不会移动真实鼠标的页面内自动 hover；必要时再使用 OpenCLI Browser Bridge 自动移动鼠标到时间 UI。业务同学不需要手动悬停时间。
- 只有登录失效、游客预览、验证码/风控、Chrome profile 不对、页面没有真实加载出帖子这类阻断问题，才需要人工介入。
- `reel/photo/watch/video/post/story` 都会先作为 FB 内容候选保留，不会因为缺父帖链接被丢弃。
- 父帖链接只作为优先去重依据；抓不到父帖时保留原始 FB 内容链接，后续再处理重复。
- 正式入表必须补齐评论区或评论回复里的引流链接，并确认最终落到外部网站。
- 评论区或评论回复里的账号自发引流链接优先级最高。详情页如果同时出现右栏广告、推荐流广告或其他外部广告链接，不能覆盖已经抓到的评论引流链接。
- 故事概要必须来自引流落地页/文章内容的中文总结，不能直接复制 Facebook 帖子正文。
- 较短帖子不会因为正文短而丢弃；只要 FB 内容链接有效，会进入待补全状态，继续补引流落地链接、文章概要和发帖时间。
- 主页滚动采集如果提示 `coverage_incomplete=true`，说明达到本次最大快照数时仍有新增候选，不能认定覆盖完整。需要从主页顶部重试，或提高快照上限继续抓。
- 正式 `--sync` 是严格最终输出：只有当前 `ready_for_output` 且通过 `quality_audit` 的记录才写入 `FB竞品帖子链接`。缺精确时间、引流链接、文章概要、互动数据、帖子类型或覆盖完整性时，记录留在 SQLite 和补抓队列中，Codex 汇报缺口并继续按 `next_commands` 补抓。
- `--sync-audit` / `--ledger-sync` 只在明确要求预览或写入不完整审计台账时使用；这类输出可以用 `待补抓：...` 标记缺口，但不是普通业务写表路径。
- 真实写入飞书前，系统会先自动检查并恢复飞书 CLI 用户身份：自动设置 `default-as user` / `strict-mode user`，遇到 `tokenStatus=needs_refresh` 会先尝试 CLI 刷新并复查。只有命令无法恢复或必须人工浏览器授权时，才会在采集/导入前停止并给出授权信息。
- 已入库记录再次导入时会优先保留更高质量字段，例如详情页确认时间、评论/回复引流链接、外部落地页、中文文章概要、互动数据和人工填写的“是否采用”。
- 飞书输出固定写入 `FB竞品帖子链接` 的 A-K 格式：`账号`, `账户类型`, `帖子链接`, `帖子类型`, `发帖时间`, `文章链接`, `故事概要`, `互动数据（点赞量）`, `浏览量`, `是否采用`, `对应站内链接`。
- 同步飞书前，需要确保 `FB竞品帖子链接` 输出表格 URL 已配置。
- 账号来源表只用于读取账号清单，不写入采集结果。
- 本工具只做 Facebook 候选采集、字段补抓、文章材料提取、中文故事概要、入库、筛选和严格同步飞书；不做营销生文、站点发布和引流帖生成。

## 具体日期采集说明

如果业务说“抓 5 月 27 日”，系统不能只按“过去 24 小时”处理。正确逻辑是：

1. 先抓目标账号附近时间段的可见帖子。
2. 主页上的 `3h/12h/1d` 这类相对时间只用来判断候选窗口和滚动边界，例如“今天”先收集 `3h`、`12h` 一类帖子，看到稳定 `1d` 可作为边界提示。
3. 对每条候选打开帖子详情页，优先通过自动时间悬停提示或 DOM 精确时间属性补齐 `posted_at`。
4. 详情页精确时间才决定是否属于目标日期；主页看起来像“今天”的帖子，可能在详情页解析后属于前一天。
5. 在详情页展开评论和回复，抓账号自己发出的引流链接，并用这个链接生成文章概要。
6. 只能看到 `19min/1h/16h/1d` 等相对时间且没有精确时间的，先进入待确认，不直接同步正式结果。
